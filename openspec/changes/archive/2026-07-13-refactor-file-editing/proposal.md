## Why

现有 `edit` / `multiedit` 主要依赖精确 `old` 文本或行号范围。模型从带行号的读取结果重建空白、换行和较大代码块时容易失败；行号在文件变化后也难以表达足够的上下文，导致重试多、误编辑风险高。

需要提供以补丁块上下文定位的主编辑路径，在保持现有调用兼容的前提下，让模型能用一个可读、可预览、可原子提交的多文件变更描述准确修改文件。

## What Changes

- 新增 `apply_patch` 写工具，接受 `*** Begin Patch` 信封格式，支持新增、更新、删除与重命名文件。
- 对更新补丁按完整上下文精确匹配；匹配缺失或不唯一时拒绝执行，并返回可操作的定位信息，绝不做模糊写入。
- 在内存中预演整个补丁，在所有路径与并发变更校验通过后才提交；失败时保持或恢复调用前的工作区状态。
- 保留 `edit` 与 `multiedit` 的既有 JSON 接口，更新工具说明与文档，优先引导模型使用带上下文的 `apply_patch`。
- 把新工具纳入现有 preview、权限、统一 diff、文件变更通知和 checkpoint 协议。

## Capabilities

### New Capabilities

- `apply-patch-tool`: 以带上下文的补丁信封安全、原子地完成多文件编辑。

### Modified Capabilities

- `fs-tools`: 注册新的写工具，并明确编辑工具的定位、原子性与失败语义。

## Impact

- 影响 `internal/tool/fs` 的工具实现、注册和测试。
- 影响 fs 工具规格、设计文档、需求文档和使用说明中的工具清单与编辑协议。
- 不新增运行时第三方依赖，也不改变既有 `edit` / `multiedit` 的入参兼容性。
